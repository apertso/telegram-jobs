# REQUIREMENTS.md — требования к Telegram Jobs Collector

## R1. telegram.csv перезаписывается при каждом запуске

**Требование:** Каждый прогон `collect.py` (через `start-telegram-jobs.bat` или
`python run.py`) начинает сбор с пустого `telegram.csv`. Файл содержит только
вакансии текущего прогона, а не накапливает данные от предыдущих запусков.

**Реализация:** `lib.reset_csv(path)` вызывается в начале `collect.py:main()`
до обработки источников. Функция перезаписывает файл пустым CSV (только заголовок
`Title,Company,Location,WorkMode,URL`).

**Внутрипрогонная дедупликация сохраняется:** если одна и та же вакансия
встречается в нескольких источниках, она добавляется в CSV только один раз
(через `lib.add_jobs` с `dedup_key`).

**Тест:** `test_lib.py` — проверяет, что после `reset_csv` файл содержит 0 строк
данных, заголовок сохранён, и последующая запись работает корректно.

## R2. Вкладки, открытые скриптом, закрываются при выходе

**Требование:** При завершении работы `collect.py` (нормальном или по ошибке)
закрываются все вкладки, открытые скриптом — в первую очередь вкладка
`connect.html`, которую MCP создаёт при подключении к Chrome через Playwright
Extension. Пользовательские вкладки НЕ закрываются.

**Реализация:**
- После `mcp.start()` вызывается `ba.list_tabs(mcp)` для нахождения вкладок с
  `connect.html` в URL. Их tabId сохраняются в `script_tab_ids`.
- В блоке `finally` `collect.py:main()` вызывается `ba.close_script_tabs(mcp, script_tab_ids)`
  перед `mcp.stop()`, что закрывает эти вкладки.

**Тест:** `test_tabs.py` — live-тест с реальным MCP: проверяет `list_tabs`
(находит вкладки), создание новой вкладки, `close_tab` (закрывает её), и
подтверждает, что закрытая вкладка исчезает из списка.

## R3. Chrome и пользовательские вкладки не закрываются

**Требование:** Скрипт не завершает процесс Chrome и не закрывает вкладки,
которые не были открыты им самим. Это позволяет пользователю продолжать работу
в браузере без прерывания.

**Реализация:** Скрипт работает только в текущей вкладке (открытой MCP).
Закрываются только вкладки с `connect.html` в URL (открытые MCP при подключении).

## R4. Фильтр по давности сообщений (TELEGRAM_SINCE_HOURS)

**Требование:** При сборе вакансий учитываются только сообщения, опубликованные
за последние `TELEGRAM_SINCE_HOURS` часов (по умолчанию 24). Сообщения старше
указанного порога отбрасываются до обращения к OpenRouter и не попадают в
`telegram.csv`. Это исключает устаревшие вакансии из результатов прогона.

**Переменная окружения:** `TELEGRAM_SINCE_HOURS` (в `.env` или `.env.example`).
Значение `0` отключает фильтр — собираются все доступные сообщения.

**Реализация:**
- `server.py` читает `TELEGRAM_SINCE_HOURS` в `SINCE_HOURS`.
- `filter_by_since(messages, since_hours)` отбрасывает сообщения, у которых
  `publishedAt` (ISO 8601) старше порога `now - since_hours*3600`. Сообщения
  без распознаваемой метки `publishedAt` сохраняются (консервативно: возраст
  не определён, поэтому данные не теряются).
- В `process_messages` фильтр применяется после валидации входа и до запроса
  к OpenRouter. Количество отброшенных сообщений возвращается в статистике
  как `filteredByTime`.
- `collect.py` агрегирует `filteredByTime` по всем источникам и выводит в
  итоговой статистике («Отфильтровано по времени»).

**Scroll-collect (заполнение окна):** `browser_agent.collect_with_scroll`
прокручивает канал в обе стороны, чтобы собрать все сообщения в окне
`TELEGRAM_SINCE_HOURS`, независимо от того, где Telegram открыл канал
(последние сообщения, первое непрочитанное, или глубоко в истории).

Цикл:
1. Начальное извлечение видимых сообщений, определение направления к
   старым/новым по `publishedAt` первого и последнего сообщения в DOM.
2. **Фаза 1** — скроллит к новым сообщениям, пока не достигнет «живого
   хвоста» (новейшая метка в пределах `SCROLL_TAIL_TOLERANCE_S` от `now`)
   или пока не перестанут появляться новые сообщения.
3. **Фаза 2** — скроллит к старым, пока старейшая видимая метка не станет
   старше cutoff (`now - since_hours*3600`), или пока не перестанут
   появляться новые сообщения (дно истории).
4. Все уникальные сообщения (по `messageId`) сливаются и сортируются
   старые→новые.

Параметры (env): `SCROLL_MAX_ITER` (по умолч. 15 итераций на фазу),
`SCROLL_WAIT_MS` (пауза после прокрутки, по умолч. 800 мс),
`SCROLL_TAIL_TOLERANCE_S` (допуск «живого хвоста», по умолч. 60 с).
При `TELEGRAM_SINCE_HOURS=0` прокрутка отключена — одно извлечение.

**Тест:** `test_lib.py` — проверяет:
- `_parse_published_at` и `_parse_iso_ts` корректно разбирают ISO с `Z` и
  со смещением, возвращают `None` для пустой/некорректной строки;
- `_detect_scroll_direction` определяет older/newer edge по publishedAt;
- `_merge_by_id` сливает сообщения по messageId без дубликатов;
- `filter_by_since` отбрасывает старые сообщения и сохраняет свежие;
- сообщения без `publishedAt` сохраняются;
- `since_hours=0` отключает фильтрацию;
- `collect_with_scroll` (с моком MCP): приземление на старых сообщениях
  (скроллит к новым и обратно), приземление на хвосте, пустой канал,
  `since_hours=0` без прокрутки;
- `process_messages` возвращает `filteredByTime` в статистике.
